# tidy-memory（指示ファイルの棚卸し・剪定）

CLAUDE.md と `.claude/rules/*.md` は**毎ターン context に前置される**。長さはそのままトークンコストであり、
重要な規約が解決済みの経緯や冗長な説明に埋もれて効きにくくなる。「読み物として正しいか」ではなく
**「常時ロードに値するか」**で判断し、肥大化を定期的に巻き戻す。

## 判断軸

各記述を次の問いで仕分ける。

1. **常時ロードが要るか？** — 毎ターン必要な普遍ルールだけを残す。特定作業時だけ要る詳細は、ファイルを
   パスで参照するだけにして本文から外すか、スキル化して呼ばれたときだけ展開させる。
2. **解決済みの経緯か？** — 「〜だったが〜に切り替えた」という背景・却下案・歴史は、結論だけ残して
   `/adr` で ADR に移す。CLAUDE.md には「何をするか」の結論と ADR への参照だけ残す。
3. **重複していないか？** — 同じ話が CLAUDE.md / 複数 rules に散っていれば、1 か所を正（single source of
   truth）とし、他は 1 行の参照に置き換える。
4. **「なぜ」が冗長でないか？** — 規約は命令形・宣言形で簡潔に。長い理由説明はリンク（ADR / 外部ドキュメント）へ。

## 手順

1. **棚卸し**: CLAUDE.md と `.claude/rules/*.md` を読み、上記 4 軸で各セクションを分類する。
   どのファイルが常時ロードされているか（CLAUDE.md からの `@import` や harness の自動ロード対象か）を
   確認し、「分割しても自動ロードされ続けるなら節約にならない」点に注意する。
2. **剪定案を提示**: 「削る / ADR へ移す / 参照に置換 / スキル化」をセクション単位で列挙し、
   おおよその削減量（行数・トークン感）とともにユーザーに示す。**破壊的な変更なので、適用前に必ず確認を取る。**
3. **経緯の退避を先に**: ADR へ移す内容があれば、先に `/adr` で ADR を起こしてから CLAUDE.md 側を削る
   （受け皿を作ってから切る）。
4. **編集を適用**: 承認された範囲で CLAUDE.md / rules を編集する。結論は残し、経緯は ADR 参照に置換する。
5. **整合性チェック**: 参照リンク（ADR・rules・skills へのパス）が壊れていないか、削除した内容が他所から
   参照されていないかを確認する。可能なら `./gradlew check` 等で副作用がないことも確かめる。
6. **コミット**（ユーザーが求めた場合のみ）: Conventional Commits 形式。例: `docs: CLAUDE.md を剪定し経緯を ADR へ移動`。

## やってはいけないこと

- **無確認で大量削除しない**。CLAUDE.md は手で育てた資産。必ず案を見せて合意を取る。
- **`/init` で再生成して上書きしない**（剪定目的では使わない）。
- **経緯を ADR に移す前に本文から消さない**（情報を失う）。
- 普遍ルールまで「冗長だから」と削らない。残すべき規約と、逃がすべき経緯を取り違えない。
