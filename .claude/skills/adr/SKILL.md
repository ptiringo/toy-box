# ADR（Architecture Decision Record）

設計・運用上の「なぜそう決めたか」を 1 決定 1 ファイルで `docs/adr/` に残す。CLAUDE.md / rules には**結論（何をするか）**だけを置き、経緯・背景・却下案は ADR に逃がすことで、常時ロードされる指示ファイルの肥大化を防ぐ。

## いつ起こすか

- 新しい設計・ツール・運用方針を**採用 / 変更 / 撤回**したとき
- CLAUDE.md / rules に「〜だったが〜に切り替えた」という**解決済みの経緯**が書かれていて、結論だけ残して経緯を移したいとき
- 後から「なぜこうなっているのか」を説明する必要が出そうな判断をしたとき

逆に、普遍的な「今守るべきルール」は ADR ではなく CLAUDE.md / `.claude/rules/` に書く。ADR は**時点の決定の記録**であり、ルールブックではない。

## 手順

1. **次の連番を決める**: `ls docs/adr/` で既存の最大番号を確認し、`+1`。4 桁ゼロ詰め（`0001`, `0002`, ...）。
2. **ファイル作成**: `docs/adr/NNNN-ケバブケースのタイトル.md`（タイトルは英語スラッグ。例: `0001-drop-github-mcp-use-gh-cli.md`）。
3. **テンプレートを埋める**（下記）。本文は**日本語**で書く（このリポジトリのドキュメント規約に従う）。
4. **索引を更新**: `docs/adr/README.md` の一覧に 1 行追加（番号・タイトル・Status）。
5. **関連 ADR をリンク**: 決定が過去の ADR を置き換える/関連する場合、相互に `[NNNN](NNNN-....md)` でリンクし、置き換えなら旧 ADR の Status を `Superseded by ADR-NNNN` に更新する。
6. **コミット**（ユーザーが求めた場合のみ）: Conventional Commits 形式。例: `docs: ADR-0001 GitHub MCP 撤去と gh CLI 直接利用を記録`。

## Status のライフサイクル

- `Proposed` — 提案中（議論の余地あり）
- `Accepted` — 採用・有効
- `Deprecated` — 非推奨だが残置
- `Superseded by ADR-NNNN` — 別の ADR で置き換え済み

確定済みの決定を記録する場合は最初から `Accepted` でよい。

## テンプレート

```markdown
# NNNN. <決定の要約タイトル>

- Status: Accepted
- Date: YYYY-MM-DD
- Deciders: <意思決定者 / 関係者>

## Context（背景・課題）

なぜこの決定が必要だったか。直面していた問題・制約・前提を書く。
検討した代替案があればここで挙げ、なぜ採らなかったかを簡潔に記す。

## Decision（決定）

何を決めたか。具体的に、命令形で。

## Consequences（結果・影響）

この決定によって何が良くなり、何を引き受けることになったか（トレードオフ）。
今後の運用上の注意点や、CLAUDE.md / rules 側に残した結論への参照。
```

## 注意

- 日付は会話の現在日付を使う（相対表現「先週」等は使わず YYYY-MM-DD に変換）。
- ADR は**確定した過去の記録**なので、後から内容を書き換えるのではなく、方針が変わったら**新しい ADR を起こして旧 ADR を Superseded にする**（履歴を残す）。
- 1 ファイル 1 決定。複数の独立した決定を 1 ファイルに混ぜない。
