# 0012. 馬のロール（種牡馬・繁殖牝馬・競走馬）は BloodHorseId 参照の別集約として表す

- Status: Accepted
- Date: 2026-06-20
- Deciders: Matsui

## Context（背景・課題）

軽種馬は一生のなかで複数のロール（役割）を担う。代表的なものに **種牡馬（Stallion）**・**繁殖牝馬（Broodmare）**・**競走馬（Racehorse）** がある。これらは別個体ではなく、**同一の馬（`BloodHorse`）がライフサイクルの段階で担う役割**である。たとえば競走馬として走った馬が引退後に種牡馬になるが、血統上の同一性は変わらない。

issue #266 でこの「ロール統一」をどう表現するかが論点になった。現状、ロールの実装状況はばらついている。

- **繁殖牝馬（Broodmare）**: `BreedingRegistration` 集約として作り込み済み（`broodmareId: BloodHorseId` で対象馬を参照、生成口 `of` は `internal`、付与は `registerForBreeding` ドメインサービスに封じ込め）。
- **種牡馬（Stallion）/ 競走馬（Racehorse）**: `data class Stallion(val bloodHorseId: BloodHorseId)` のような中身のないスタブ。

ロールの表現方式として 2 案を検討した。

- **案A: `BloodHorse` 本体にロール状態を内包する**。`BloodHorse` に「繁殖供用中」「競走馬登録済み」などの状態やロール固有の属性（繁殖登録番号など）を持たせる。
  - 却下理由: ①`BloodHorse` が肥大化し、血統上の identity（不変の核）という責務がぼやける。②ロールごとに固有の登録番号・ライフサイクル（繁殖登録番号、供用停止状態 #318、競走成績など）を持つため、状態が増えるたびに集約が膨らむ。③イミュータブル集約（[ADR-0009](0009-immutable-aggregates.md)）では状態遷移ごとに巨大な `copy` が必要になる。④何より、既に実装済みの `BreedingRegistration`（別集約方式）と矛盾する。
- **案B: ロールごとに `BloodHorseId` を参照する別集約として表す**。`BloodHorse` は血統上の identity に集中させ、各ロールは独立した集約として `BloodHorseId` で対象馬を指す。
  - 採用。理由は後述。`BreedingRegistration`（繁殖牝馬ロール）が既にこの方式で実装されており、確立した先例がある。本プロジェクトの規約（[architecture.md](../../.claude/rules/architecture.md)、jMolecules `JMoleculesDddRules`）が「他集約への参照は ID 値クラス経由のみ」を ArchUnit で強制しているため、別集約 + ID 参照は規約とも自然に整合する。

あわせて、公式資料（JAIRS = 日本軽種馬登録協会の登録規程・実施基準）を精読し、**ロールごとに権威ソース（authority）が異なる**ことが判明した。これは設計に効く重要な事実である（典拠は issue #266 のコメントに整理済み）。

- JAIRS が行う登録は **血統登録**（血統及び個体識別を明らかにする登録）と **繁殖登録**（繁殖に供する馬の繁殖成績を明らかにする登録）の **2 種類のみ**。
- 繁殖登録は「血統登録証明書を持つ既存個体」に対して行う追加登録であり、別個体を作らない（繁殖登録原簿 様式第2号は血統登録番号と繁殖登録番号が同一様式に並存）。種牡馬（繁殖用 雄）・繁殖牝馬（繁殖用 雌）は**繁殖登録のタイミングで確定するロール**で、`BloodHorse`（血統登録済）を前提条件とする。
- 一方、**競走馬登録（出走資格）は JAIRS の管轄外**であり、JRA（中央競馬）・地方競馬（NAR）の権威に属する。原簿には「中央又は地方の別」として注記される程度。つまり Racehorse ロールの不変条件は JAIRS 規程からは導けず、ドメインモデルとして自前で定義する必要がある。

## Decision（決定）

- **馬のロール（種牡馬・繁殖牝馬・競走馬）は、`BloodHorse` 本体に内包せず、`BloodHorseId` を参照する独立した集約として表す。** 既存の `BreedingRegistration`（繁殖牝馬ロール）をこのパターンの先例とし、種牡馬・競走馬も同じ型に揃える。各ロール集約は以下の約束を守る。
  - **対象馬は `BloodHorseId` で参照する**（集約オブジェクトを直接持たない。`JMoleculesDddRules` / ArchUnit で強制）。
  - **生成口（`of` 等）は `internal`** にし、ロール付与の前提条件を検証するドメインサービス（`service/` のトップレベル関数）からのみ呼べるよう封じ込める（[ADR-0010](0010-confine-aggregate-creation-to-domain-service.md) のパターン）。ロール付与は「血統登録済み（`BloodHorse` が存在する）」を最低限の前提条件とする。
  - 集約は**イミュータブル**に保ち、状態遷移（供用停止 #318 など）は新インスタンスで表す（[ADR-0009](0009-immutable-aggregates.md)）。
- **ロールごとの権威ソースの違いを明示する。** 血統登録・繁殖登録は JAIRS が権威であり、種牡馬・繁殖牝馬ロールの不変条件は JAIRS 規程に典拠を持つ。**競走馬（Racehorse）ロールは JAIRS の管轄外**（JRA / NAR が権威）であり、その不変条件は JAIRS 規程からは導けない。競走馬登録のモデルは、当面この権威境界を踏まえて最小限の前提条件（血統登録済み・馬名登録済み等）でドメインモデルとして定義し、外部の権威に依存する属性（出走資格の確定など）は本コンテキストの責務外として切り離す。

## Consequences（結果・影響）

- `BloodHorse` は血統上の identity（不変の核）という単一責務に集中でき、ロールが増えても肥大化しない。ロール固有の属性・ライフサイクルは各ロール集約に閉じる。
- ロール追加の判断基準が「`BreedingRegistration` と同じ型に揃える」に統一される。新しいロール（競走馬: #266、種牡馬の集約化）は、別集約 + `BloodHorseId` 参照 + `internal of` + 付与ドメインサービス、というテンプレートを写経して実装できる。
- 「同一の馬がロールを担う」は `BloodHorseId` の共有で表現される。別個体は作らないという JAIRS の制度（繁殖登録は血統登録個体への番号付与）とも整合する。
- 競走馬ロールは権威ソースが JAIRS 外であることを引き受ける。JAIRS 規程に典拠を求められないため、不変条件は薄く、ドメインモデルとしての定義は探索的になる。出走資格・競走成績など外部権威に属する情報を将来扱う場合は、本コンテキストの境界を越えるものとして別途設計する（[ADR-0012] の射程外）。
- スタブのままだった `Stallion` / `Racehorse`（`BloodHorseId` のみ保持）は、この方針に沿って集約へ昇格する。競走馬登録（#266）を最初の実装とし、種牡馬ロールの集約化（現状は種付証明書 `Covering` 文脈で部分的に扱われている）は別途整理する。
- 繁殖ロールのライフサイクル終端（繁殖供用停止: #318）は、本決定により繁殖ロール集約のイミュータブルな状態遷移として収容する余地が生まれる。
