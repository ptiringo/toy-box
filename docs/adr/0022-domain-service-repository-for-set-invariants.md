# ADR-0022: ドメインサービスは集合制約の検証に限りリポジトリポートを受け取ってよい

- ステータス: Accepted
- 日付: 2026-06-23
- 関連: Issue #370, PR #398, ADR-0014, ADR-0009, [.claude/rules/architecture.md](../../.claude/rules/architecture.md), [.claude/rules/testing.md](../../.claude/rules/testing.md)

## コンテキスト

これまで本プロジェクトは「ドメイン層は純粋（リポジトリ I/O を持たない）／リポジトリからの引き当て（coordination）はアプリケーション層」という役割分担を採ってきた（architecture.md）。ドメインサービス（`service/` のトップレベル関数）は解決済みの集約を引数で受け取る純粋関数として書き、テストは Fixture のみ・モック不要（testing.md）という性質を得ていた。`registerFoal` / `confirmRaceResult` がその例。

#370 で「繁殖成績は『繁殖牝馬 × 繁殖年』で一意」という不変条件（同一年の重複記録の禁止）を種付記録に追加する必要が生じた。当初はこの一意性検証を自己検証ファクトリ `BreedingResult.create` の引数 `existingForYear: BreedingResult?` に持たせたが、これは「不在を確認するために存在物（あるかもしれない既存レコード）を引数で受ける」という意味論の歪みを生んだ。

一意性のような制約は、**単一インスタンスの構築時不変条件ではなく、既存レコード集合への問い合わせがなければ判定できない「集合制約」**である。これは architecture.md がドメインサービスに与える定義（単一集約の構築ではない、集約／集合をまたぐドメインロジック）に合致する。そこで検証をドメインサービス `recordCovering` へ移したが、ここで分岐が生じた:

- **案A（純粋サービス）**: ドメインサービスは純粋のまま、既存成績の引き当ては app 層が行い結果を引数で渡す。既存規約に一貫するが、`existingForYear` 引数の意味論の歪みがサービス側に残る。
- **案B（リポジトリ注入）**: ドメインサービスがリポジトリポートを引数で受け取り、自分で引き当てて集合制約を検証する。操作が自己完結し意味論の歪みが消える。Evans 的にもドメインサービスがリポジトリを使うのは王道。

案B はオニオン依存規則・ArchUnit には抵触しない（リポジトリポートは `domain.*.model` に属し、`service → model` の依存方向に反しない。`./gradlew check` も緑）。抵触するのは2つの**散文規約**のみ:

1. architecture.md「リポジトリからの引き当て（coordination）はアプリケーション層」
2. testing.md「ドメイン層は Fixture で実物を組み、モックしない」（案B ではドメインサービステストがリポジトリポートをモックする）

「ドメインサービス1個だけ流儀を変える」と無条件に許すとコードベースが不揃いになり、純粋ドメイン層の利点（I/O を app 端に集約・テストがモック不要）を無秩序に失う。そこで**案B を採用しつつ、ドメインサービスがリポジトリを使ってよい条件を厳密に定める**。

## 決定

ドメインサービス（`service/` のトップレベル関数）は、**集合制約のドメイン不変条件を検証するために必要な読み取り引き当てに限り**、リポジトリポートを引数で受け取ってよい。

許可の条件（すべて満たすこと）:

1. **目的が集合制約の検証であること**。一意性・件数上限など「他に同条件のレコードが存在しないか」を既存レコード集合に問い合わせないと判定できないドメイン不変条件の検証に用いる。単一インスタンスで完結する構築時不変条件は引き続き**自己検証ファクトリ**が担い（ADR-0014）、リポジトリを取らない。
2. **読み取り専用であること**。ドメインサービスは `findBy...` 等の引き当てのみ行い、`save` 等の書き込みはしない。永続化はアプリケーション層が行う。
3. **リポジトリ「ポート」を引数で受け取ること**。ドメインサービスはトップレベル関数のまま（`object` / `class` でラップしない・Spring 非依存）。フィールドインジェクションや具象実装への依存はしない。ポートは `domain.*.model` のインターフェースに限る。

許可しないもの（引き続きアプリケーション層の責務）:

- 入力参照の解決（コマンドの ID → 集約のロード）と、その不在に対する応答（`〜NotFound` → 404 / 422）。
- 操作対象集約のロード、結果の保存（`save`）。
- 複数ユースケース手順のオーケストレーション。

テスト規約の例外: 上記条件でリポジトリポートを取るドメインサービスのテストでは、そのポートをモックしてよい（testing.md「ドメイン層はモックしない」の限定的な例外）。

参考実装は `recordCovering`（`domain/horseracing/service/breeding/RecordCovering.kt`）。`broodmareRegistration` / `stallionRegistration` 等の解決済み集約に加えて `BreedingResultRepository` を受け取り、同一繁殖牝馬・同一繁殖年の既存成績を引き当てて一意性（`AlreadyRecordedForYear`）を検証し、満たせば構築時不変条件（登録ロール）の検証を `BreedingResult.create` に委譲する。

## 理由

- **意味論の整合**: 集合制約は集合への問い合わせが本質。引き当てをサービス内に置くと「不在確認のために存在物を引数で受ける」歪みが消え、操作が自己完結する。
- **役割の明確な分離**: 「構築時不変条件＝ファクトリ（ADR-0014）／集合制約＝ドメインサービス（本 ADR）／入力解決・永続化・オーケストレーション＝app 層」と、検証の性質で担い手が決まる。
- **規律の維持**: 「読み取り専用・ポート引数・集合制約の検証に限る」という枠を課すことで、ドメインサービスへの無制限なリポジトリ流入（書き込み・オーケストレーションの染み出し）を防ぐ。純粋でよいサービス（`registerFoal` 等）は純粋のまま据え置き、強制的な全面移行はしない（＝「取ってもよい」許可であって義務ではない）。
- **オニオンに反しない**: リポジトリポートは `domain.*.model` にあり、`service → model` の依存方向に従う。ArchUnit でも違反にならない。

### 案A（純粋サービス＋app 引き当て）を採らなかった理由

案A は既存の純粋ドメイン層と一貫するが、`existingForYear` 引数の意味論の歪み（不在確認のための存在物受け渡し）がサービスのシグネチャに残り、集合制約の検証が「app が引いて service が判定する」と二箇所に割れる。集合制約の検証主体をサービスに一本化し読み口を自然にする案B を、条件付きで採る方が設計の説明力が高いと判断した。

## 影響

- architecture.md のレイヤー依存ルール（domainService 節）に、本 ADR の許可条件を結論として追記する。
- testing.md に、集合制約検証のためリポジトリポートを取るドメインサービスのテストではポートをモックしてよい旨の例外を追記する。
- `recordCovering` は `BreedingResultRepository` を引数で受け取り、`RecordCoveringUseCase`（app 層）は同年の既存成績の事前引き当てを手放してサービス呼び出しと永続化に専念する。
- 今後、集合制約（一意性・件数上限など）をドメイン不変条件として表すドメインサービスは、本 ADR の条件下でリポジトリポートを取ってよい。それ以外のリポジトリ利用は引き続き app 層に置く。
