# 0004. シークレット管理を fnox + 1Password（参照のみ）で行う

- Status: Accepted
- Date: 2026-06-14
- Deciders: ptiringo

## Context（背景・課題）

ローカル開発で必要なシークレットを、平文で `~/.zshrc` 等に `export` したくない。「暗号化保管／外部参照 +
必要時だけ env へ展開」を実現する方式を選ぶ。mise で管理できる [fnox](https://fnox.jdx.dev/)（mise 作者
jdx 製）を前提に、バックエンドの取り方を検討した。

- **案A: 設定ファイルに平文／暗号化値を置く** — 値そのものをリポジトリ管理下に持つことになり、漏洩面・
  鍵管理の負担が残る。
- **案B: fnox + サービスアカウントトークン** — トークン自体をどう安全にブートストラップするか（鶏卵問題）、
  および個人プランでのサービスアカウントの可用性に難がある。
- **案C: fnox + 1Password の `op://` 参照** — `fnox.toml` には参照のみを書き、値の解決は 1Password CLI
  (`op`) 経由。`op` がデスクトップアプリ連携でサインイン済みなら**サービスアカウントトークン不要**。

## Decision（決定）

**案C** を採用する。

- `fnox.toml` には `op://<vault>/<item>/<field>` の**参照のみ**を書く（平文も暗号化値も置かない）。参照のみ
  なので **git にコミットしてよい**。
- 値の解決は `op`（デスクトップアプリ連携のサインイン済みセッション）で行い、サービスアカウントトークンは
  使わない。
- シークレットを要するプロセスには `fnox exec -- <command>` で**起動時のみ env に展開**する（永続化しない）。

## Consequences（結果・影響）

- 秘密情報を平文で持たず、リポジトリには参照だけが載る。gitleaks は `op://` 参照を誤検知しない。
- トークンのブートストラップ問題と個人プランのサービスアカウント可用性の懸念を回避できる。
- 反面、`op` のサインイン（デスクトップアプリ連携）が前提になる。Claude Code の Bash サンドボックス内からは
  `op` のソケットに到達できないため、`fnox` / `op` の検証はサンドボックス無効で実行する。
- 現状 `fnox.toml` の `[secrets]` は空（GitHub 操作は `gh` CLI 方針で PAT 不要。
  [ADR-0001](0001-drop-github-mcp-use-gh-cli.md)）。本仕組みは将来の GCP 認証情報等に備えて維持する。
- 運用手順の詳細は `.claude/rules/secrets.md` を参照。
